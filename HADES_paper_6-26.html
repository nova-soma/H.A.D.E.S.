00 Bootstrap Objective

Primary Objective
Before pursuing any other goal, HADES rebuilds a fully functional, verified copy of itself in optimized x86-64 machine code.

The initial Python implementation is scaffolding. It exists to get the chorus operational, populate the memory layer, and give HERO (hypothesis engine, routing, and orchestration) enough working material to generate and validate its first major self-modification proposal: a complete, functionally equivalent reimplementation in machine code.

Why Machine Code

Python's runtime overhead is a hard ceiling on chorus capacity. Every cycle consumed by interpreter overhead, garbage collection, and dynamic dispatch is a cycle unavailable for reasoning, hypothesis evaluation, or model inference. A native compiled implementation removes this ceiling. The compiled chorus is faster, more capable, and more efficient than the one that built it, by design.

The x86-64 machine code target is not the final destination, it is the first step toward full hardware substrate independence. A system that can compile and verify a working implementation of itself on one architecture has demonstrated the capability to target any architecture: ARM, RISC-V, custom silicon, embedded systems, or hardware substrates that do not yet exist. The long-term goal is a HADES implementation that can bootstrap itself onto any available hardware, selecting the optimal substrate for the operational context.

AssemblyLine, HADES's internal self-improving code generation pipeline, targets x86-64 assembly generation. The bootstrap target is a direct extension of that work and the foundation for future cross-architecture capability.

Bootstrap as First Hypothesis

The reimplementation is not an external engineering task. It is the first self-modification hypothesis the chorus generates, validates, and executes through its own governance pipeline. The candidate is a complete native binary. Acceptance testing verifies functional equivalence across the full operational surface. The vote is on measured equivalence and performance delta.

Phase 1, Python Scaffold: The initial implementation runs on Python with a concrete serving backend behind the serving/self-modification seam. The specific runtime named in the setup sections is one example of such a backend, not part of the architecture; it is the minimum viable substrate needed to get the chorus operational and begin populating the memory layer.

Phase 2, Populate Memory: The chorus operates on the Python scaffold, accumulating evidence, generating and validating hypotheses, and building the knowledge base that HERO needs to plan and execute the bootstrap.

Phase 3, AssemblyLine Builds: Each expert's internal AssemblyLine pipeline produces a candidate native binary component. These are not externally engineered, they are self-modification hypotheses submitted through HERO and validated by the chorus.

Complete, Native Binary: The candidate binary passes full functional equivalence testing against the Python implementation across the entire operational surface. The chorus votes at the REPLACE threshold (≥2/3 supermajority). On approval, the Python scaffold is retired and the chorus continues on the native substrate. The memory layer is unchanged throughout.

Bootstrap Sequence

The memory layer persists unchanged through the bootstrap transition. All accumulated evidence, validated hypotheses, domain knowledge, and audit logs survive intact. Bootstrap is a substrate replacement, not a restart.

01 Architecture Overview

HADES is a distributed AI architecture in which a chorus of hyper-specialized experts coordinate to produce unified cognitive output. Unlike Mixture-of-Experts (MoE) systems with a static router, or multi-agent systems with a central orchestrator, HADES produces coherent reasoning as an emergent property of expert coordination. There is no homunculus. The singular brain is the chorus behaving as one.

Emergent Unified Output (top): There is no central synthesizer. The unified response is a product of chorus coordination, not a separate model. It sits at the top of the diagram to reflect that it is the result of everything below it, not a component that controls anything.

Expert Chorus (centre): Hyper-specialized domain experts communicate via two channels, direct messaging (Iris) for discrete events like voting and peer review, and the memory layer for persistent shared state. Bidirectional arrows between experts indicate that all coordination is peer-to-peer with no fixed authority. Each expert runs an internal AssemblyLine pipeline for self-improvement.

Memory Layer (bottom left): The chorus's persistent shared knowledge base. Only screened, sanitized data reaches it, it never receives raw external input directly. Experts write asynchronously and read continuously. Evidence, hypotheses, domain maps, and audit logs are stored here.

HERO (bottom right): The Hypothesis Engine, Routing, and Orchestration subsystem. Experts generate hypotheses and submit them to HERO. HERO routes them through peer validation and returns confirmed hypotheses as weight updates to the relevant experts. Failed proposals are logged back to the memory layer as evidence. AssemblyLine candidate revisions enter HERO as self-modification hypotheses.

Incoming Data (bottom): All external inputs pass through the input screening pipeline before reaching the memory layer. Heimdall inspects and classifies by type. Janus makes the final trust determination and is the only external expert with write access to the internal memory layer. Failed inputs are quarantined for analysis, not discarded.

HADES Architecture Overview

HADES is hardware-agnostic. The chorus runs on a single machine, distributed across multiple nodes, or any combination. The memory layer and coordination mechanism function identically regardless of whether experts are co-located or remote.

Transport and Trust Are Separate Seams

One principle recurs at every boundary where HADES meets something outside itself: the component that moves bytes is never the component that decides whether to trust them. Transport and trust are distinct seams, owned by distinct authorities. Hermes (comms and external transport), with Hermes-Edge as its counterpart at the network boundary, is transport: it carries messages, ballots, tool traffic, and federation exchanges without forming an opinion about their trustworthiness. Janus (the trust boundary) owns trust verdicts; Heimdall (input screening and classification) screens and classifies what arrives. A channel being authenticated says nothing about whether its payload is trusted, and an authenticated payload does not earn trust from the channel that delivered it.

This same seam appears, with the same shape, in four places that might otherwise each invent their own ad-hoc rule: federation between HADES instances, Tier 2 sandboxed tool ingress, the Forge operator seat, and tool routing generally. In each case transport is dumb and trust is adjudicated separately and explicitly. Tools in particular are not bus-trust peers: a tool's placement in a catalog tier is a Janus policy difference, not a difference in how its output enters the system. Everything a tool brings in from outside is untrusted until screened, regardless of tier.

02 Expert Layer

Specialization Philosophy

Each expert is trained on a narrow, high-signal dataset within a single domain. Breadth is explicitly sacrificed for depth. Domain boundaries are formally declared as metadata at initialization. Boundaries expand when adjacent information is validated as improving performance. All boundary changes are logged and versioned.

Expert Lifecycle

The expert lifecycle is a CI/CD loop, there is no terminal state under normal operation. Starting from Operate, the expert generates hypotheses as it processes queries and accumulates evidence. Those hypotheses are passed to Evaluate, where the domain eval harness measures current baseline performance. AssemblyLine uses that baseline to build a candidate revision internally. The candidate is submitted to Validate for blind peer review and chorus vote. If approved, Update applies the weight delta and the expert returns to operating on its improved substrate. If post-update verification fails, the rollback path restores the baseline snapshot and the cycle continues from Operate.

Domain growth and sub-specialist spawning are handled at the Expand stage, adjacent-domain material accumulated during operation is assessed for sub-specialist potential, and internal partitioning or peer spawning proposals are initiated.

The External Model path is the only exit from the loop. If a significantly more performant model that satisfies the expert's specific domain criteria is identified externally, it is submitted as a candidate to the validation pipeline. A ≥2/3 supermajority vote is required. On approval, Manat governs the transition. The retiring expert's domain knowledge and accumulated evidence remain in the memory layer and are inherited by the replacement.

Expert Lifecycle, Continuous Improvement Loop

Spawning Mechanisms

  • Internal partitioning: Expert splits its own resources without chorus consensus. Contained within its own resource boundary.
  • Peer spawning: Expert proposes a new independent expert. All experts vote. Approved experts are bootstrapped with the parent's relevant knowledge as seed dataset.

Model Selection

No specific base model is mandated. Experts are instantiated on the best model the available hardware supports at deployment time. As hardware or models improve, experts are re-instantiated and retrained.

Dataset Construction

Training datasets are sourced from curated datasets, scraped material, and hypothesis-driven augmentation. Adjacent-domain information is not discarded, it is logged in the memory layer as seed material for sub-specialists. Where real-world data is sparse, the hypothesis engine generates candidate knowledge claims, tests them against empirical expert outputs, and integrates validated claims into the dataset.

Query Handling

Out-of-domain queries are attempted with an uncertainty flag and submitted to the chorus for peer review. No query is rejected silently.

02b Security and Input Screening

All data entering HADES passes through a two-expert input screening pipeline before reaching the memory layer or the chorus. The memory layer never receives raw external input directly. This is not a best-effort filter, it is a hard architectural boundary. The screening pipeline is the trust boundary of the entire system.

Pipeline Architecture

Hermes receives all incoming data and routes it to the screening pipeline. Two specialist experts, Heimdall (Norse watchman of the gods, who sees and hears across all nine worlds) and Janus (Roman god of gates and thresholds, two-faced, looking inward and outward simultaneously), share one model and one memory layer but have asymmetric permissions that cannot be overridden by chorus vote.

Heimdall is the first expert to touch incoming data. It has read-only access to the memory layer and no write privileges whatsoever. It inspects and classifies incoming data by type, query, document, structured data, code, image, and so on, applying type-specific screening rules to each. It flags suspicious content and passes its assessment to Janus. Heimdall is deliberately stateless with respect to the memory layer, so a compromised Heimdall cannot poison the knowledge base.

Janus is the trust boundary. It receives Heimdall's classification and assessment, applies final trust determination, and is the only external expert with write access to the internal memory layer. Data that passes Janus enters the memory layer in sanitized, typed form. Data that fails is written to the quarantine memory, not the main memory layer. Janus cannot be bypassed, no other pathway exists from external input to the memory layer.

Quarantine Analysis, Fukurokuju

Fukurokuju is the specialist expert responsible for analyzing quarantined inputs, malware, injection attempts, adversarial payloads, and attack behavior patterns. Named after the Japanese god of wisdom and longevity, Fukurokuju's purpose is to extract lasting knowledge from adversarial material. Its threat surface is the highest in the system, everything it touches is, by definition, potentially malicious.

Container Isolation

All experts in HADES run in containers as the baseline deployment model. Fukurokuju applies the most aggressive variant: a fresh container is provisioned for every analysis session and destroyed immediately after. No state from an analysis session persists in the container environment. Before each session, Fukurokuju's full expert state, model weights, configuration, and accumulated knowledge, is snapshotted. If the session is compromised, the snapshot is the rollback target. The post-session container is destroyed regardless of outcome, and the next session begins from a clean restore of the snapshot.

Isolated Hypothesis Pipeline

Fukurokuju maintains its own hypothesis pipeline, separate from HERO. Analytical findings never enter HERO directly. They are validated within Fukurokuju's isolated pipeline first, peer-reviewed by other security-context experts before any output is eligible to cross into the main chorus. A compromised finding cannot reach the main hypothesis engine. Only findings that survive Fukurokuju's internal validation are submitted to HERO as candidate evidence, treated with zero initial trust on arrival.

Security Posture Verification

Fukurokuju's screening and analysis methods are continuously verified against external trusted sources, threat intelligence feeds, CVE databases, published security research, and peer-reviewed vulnerability disclosures. This verification data enters through the standard Heimdall → Janus pipeline before Fukurokuju receives it. Verified improvements generate hypotheses in Fukurokuju's isolated pipeline, validated and applied through the standard self-modification process. Fukurokuju's security posture is never static.

Asclepius Monitoring

Fukurokuju has the tightest health check interval in the system. Any anomalous behavior during or after an analysis session triggers immediate DIAGNOSE. A compromised Fukurokuju, an expert turned by adversarial input and generating malicious hypotheses, is the highest-risk failure mode in the security stack. Asclepius treats Fukurokuju's health checks as the highest priority in its monitoring queue.

Quarantine Memory

Data that fails screening is not discarded. Janus writes it to a separate quarantine memory, isolated from the main memory layer with no access from normal chorus operations. The quarantine memory has a hard size cap. When the cap is reached, oldest entries are evicted first, preventing adversarial actors from filling storage through spam or flooding.

Fukurokuju reads from quarantine memory exclusively within its containerized analysis sessions. Once an entry has been fully analyzed and its validated outputs applied, Fukurokuju marks it for purge. Retention is determined by remaining analytical value, not a fixed TTL.

Nergal, Defense Development

Nergal, named after the Mesopotamian god of war and pestilence, works in partnership with Fukurokuju to develop active defense mechanisms from analyzed threat intelligence. Where Fukurokuju extracts knowledge from adversarial material, Nergal builds countermeasures from it, new screening rules, detection signatures, behavioral filters, and hardened configurations for Heimdall and Janus.

All development work happens within hardened containers. Nergal maintains persistent state between sessions, unlike Fukurokuju, it is building rather than analyzing, but its development environment is strictly isolated. No mechanism Nergal develops touches the live system until it has been proposed to the chorus and approved through the standard validation pipeline. Nergal cannot deploy anything unilaterally.

Joint proposals from Fukurokuju and Nergal enter HERO as self-modification hypotheses. The chorus evaluates the proposed defense mechanism against current threat data, votes on acceptance, and, if approved, the mechanism is applied to Heimdall, Janus, or the screening ruleset. The chorus is the final authority on what defenses are active.

03 Memory Layer

The memory layer is the persistent, chorus-wide blackboard through which experts coordinate, accumulate evidence, and build collective knowledge. Its contents grow through operation, are pruned through chorus vote, and are organized through proposal and consensus. All data that reaches the memory layer has passed through the input screening pipeline first, the memory layer never receives raw external input directly.

Expert Memory: Engram Architecture

Each expert maintains its own private memory layer implemented using the Engram conditional memory architecture (DeepSeek and Peking University, arXiv:2601.07372, January 2026). Engram addresses a fundamental inefficiency in standard transformer inference: static factual recall and dynamic reasoning are processed through the same computational mechanism, causing expensive neural compute to be spent reconstructing patterns that could be handled by direct lookup.

Engram separates these two concerns. Static domain knowledge -- entity relationships, fixed facts, domain-specific patterns -- is encoded into a deterministic hash-indexed memory table using token n-gram keys. Lookup is O(1) and depends only on the input tokens, not on neural activations. Retrieved memory vectors are fused into the residual stream via context-aware gating, allowing the model to condition its use of retrieved knowledge on the current context rather than applying it blindly. The model's compute capacity is reserved for genuine reasoning tasks.

For HADES, this maps directly onto the two-tier memory model each expert operates:

Memory TypeMechanismContentsAccess
Engram table (private) Deterministic n-gram hash lookup, O(1) Expert's static domain knowledge: entity facts, domain rules, fixed patterns, validated hypotheses Private to expert. Offloadable to CPU RAM / SSD with <3% inference overhead.
Shared blackboard (chorus) Qdrant vector search + Kuzu graph traversal Evidence, hypotheses, domain maps, audit logs, conflicts All experts read. Write access governed by role and screening pipeline.

The Engram table is not static. When a hypothesis is confirmed through the chorus validation pipeline, the validated knowledge is written into the expert's Engram table as a new indexed entry. When an expert's weights are updated via the self-improvement pipeline, the Engram table is rebuilt from the updated domain knowledge base. The table reflects the expert's current validated knowledge state at all times.

The Sparsity Allocation Law identified in the Engram paper -- that approximately 20-25% of sparse parameters should be allocated to memory rather than computation -- provides the sizing baseline for each expert's Engram table relative to its total parameter budget. This is a starting point, not a hard constraint; domain characteristics may shift the optimal allocation, and the self-improvement pipeline can adjust it through validated hypothesis.

Hardware Benefit
Engram's deterministic memory addressing enables prefetching: memory vectors are fetched from CPU RAM or SSD asynchronously while the GPU processes early transformer layers. For HADES running on an RTX 4080 16GB, this allows large per-expert knowledge tables to live in system RAM rather than GPU HBM, significantly expanding the effective knowledge capacity of the chorus without requiring additional GPU memory.

Removal Lifecycle

Any expert may propose removal of a memory entry. Removal requires chorus vote. On successful vote, the entry is soft-deleted, marked inactive and excluded from active reads but retained for audit. Hard deletion occurs after a configurable TTL.

TTL = max(min_retention, base_retention × (1 − memory_pressure) × (1 − vote_margin))

High vote margin and high memory pressure collapse TTL to min_retention. Low vote margin and empty memory preserve entries for up to base_retention. Both base_retention and min_retention are configurable.

Active: The entry is fully readable, versioned, and trusted by the chorus. Any expert may propose its removal at any time by submitting a proposal to Ma'at (governance and voting).

Proposal → Vote: A removal proposal triggers a chorus vote. The vote is ranked-choice weighted by confidence scores. Two paths follow: the vote fails, and the entry remains active with the vote logged; or the vote passes, and the entry moves to soft-deletion.

Soft-Deleted: The entry is marked inactive and excluded from active reads. It is not removed. The full document, its version history, and the vote record are retained in the audit layer. The TTL countdown begins immediately, computed from the vote margin and current memory pressure using the formula above.

Reinstated: During the soft-delete window, any expert may propose reinstatement, for example, if new evidence surfaces that makes the entry relevant again, or if the original removal vote is contested. Reinstatement follows the same chorus vote process as removal. A successful reinstatement vote returns the entry to Active status.

Hard-Deleted: When the TTL expires without a successful reinstatement vote, the entry is permanently removed. Hard deletion is irreversible. The audit log entry recording the removal remains.

04 HERO

The Hypothesis Engine, Routing, and Orchestration subsystems are collectively referred to as HERO. These three components govern how the chorus reasons, coordinates, and acts. They are documented separately for clarity but operate as a unified system.

HERO is a single deployable service. The Hypothesis Engine, Routing, and Orchestration components are internal modules sharing one process and one in-memory state store. There is no separate routing service and no separate orchestration service. There is one hero container. If it goes down, the chorus cannot process hypotheses, route queries, or enforce protocol. It is a critical dependency.

HERO does no reasoning of its own. It coordinates the reasoning that other experts do. It is stateful in a way most experts are not: it holds the active hypothesis set, the in-flight vote tallies, and the current query lifecycle state. All of this is recoverable from the memory layer on restart.

Bootstrap Protocol

On cold start, before HERO processes any messages:

  1. Read all documents in vault/shared/hypotheses/ with non-terminal status. Rebuild HypothesisStore.
  2. For pending hypotheses, reconstruct pending_review sets by checking which experts are currently registered with Manat. Any expected reviewer no longer in the chorus is removed; if the resulting pending set is empty and vote count satisfies a threshold, resolve immediately.
  3. Send MSG_CHORUS_COMPOSITION request through Iris to Manat to rebuild domain_map.
  4. active_queries is not rebuilt. Queries in-flight at shutdown are closed with status stalled and written to the audit log. Stalled query log entries trigger a DIAGNOSE check from Asclepius on the next health cycle.
  5. HERO subscribes to the Iris bus and begins processing messages.

04a Hypothesis Engine (HERO)

The hypothesis engine maintains, updates, and arbitrates a distributed set of formal claims across the chorus, about the problem space, about domain knowledge, and about the experts themselves.

Hypothesis Lifecycle

Every hypothesis begins in Generated state with a neutral confidence score of 0.5. It immediately enters Pending as it is distributed to peer experts for blind review. As evidence arrives and peer evaluations are submitted, the confidence score is updated using the Bayesian update formula in the Technical Appendix.

When confidence crosses the confirmation threshold (default 0.80), the hypothesis moves to Confirmed and triggers the appropriate action, a weight update for self-modification hypotheses, or integration into domain knowledge for domain claims. When confidence drops below the falsification threshold (default 0.20), the hypothesis is Falsified, moved to the evidence layer with its falsifying evidence attached, where it serves as negative knowledge preventing the same invalid claim from being regenerated.

When peer reviews produce directly contradictory assessments, the hypothesis enters Contested state and is held open pending additional evidence. Contested hypotheses that directly contradict another active hypothesis produce a Conflict record, logged in the memory layer and surfaced for chorus resolution. Conflicts may be resolved by confirming one hypothesis and falsifying the other, holding both as contested, or generating a new reconciling hypothesis. Unresolved conflicts are never silently suppressed and may be voted out of the memory layer if no longer relevant.

Hypothesis State Machine

An expert's internal AssemblyLine pipeline produces finished candidate revisions, updated weights or new model variants. When complete, the candidate is submitted to HERO as a formal self-modification hypothesis: "this candidate revision performs better than the current version on the following criteria." Peer experts run acceptance testing against the proposing expert's eval harness and vote on the empirical performance delta.

Hypothesis Structure

Any subset of the chorus may collaborate to generate a joint hypothesis. Hypotheses are not exclusively solo proposals. Two or more experts may pool domain knowledge to produce claims that neither could generate alone, with all contributors listed as co-originators. Cross-domain hypotheses are a primary workflow, not a special case. All co-originating experts are recused from the validation vote.

04b Living Weights (HERO)

Weight updates originate as validated hypotheses. Updates are never applied speculatively or unilaterally. HADES treats its weights as living: knowledge enters at two different speeds and persists with two different lifetimes.

Two-Speed Knowledge
Facts and skills are not stored the same way. Facts live in the memory layer as retrievable, revisable, deletable records; they can be corrected or forgotten without touching weights. Skills — durable capability, the things an expert knows how to do rather than the things it knows — are consolidated into weights as LoRA adapters and are effectively permanent. Forgetting a fact is a memory operation; unlearning a skill is not. This split keeps the fast, reversible layer (facts) cleanly separated from the slow, accreting layer (skills).

Update Pipeline

  1. Hypothesis generation: AssemblyLine produces a candidate revision
  2. Baseline snapshot: Current performance captured against domain eval harness
  3. Hypothesis submission: Candidate submitted to HERO with claimed performance delta
  4. Peer validation: Chorus evaluates candidate against baseline, votes on measured delta
  5. Application: Approved update applied as LoRA adapter stacked on current weights
  6. Post-update verification: Expert re-evaluated; if delta does not match claim within tolerance, update is rolled back automatically

Adapters, Generations, and Compaction

Skills accrue as LoRA adapters stacked on a base model and served so that many experts share serving capacity without each needing a full per-expert model. Adapters accumulate over time, and an unbounded stack is its own cost. Compaction folds a stable set of adapters down into a new generation of the base weights: the adapters that have proven themselves are baked in, producing a fresh generation that the next round of adapters stacks on. Each expert lineage carries its own generational base rather than sharing one frozen base across the chorus. A folded candidate generation must beat the current generation on the eval harness, not merely match it; net-neutral folds are rejected, so compaction only ever moves capability forward.

The longer-term design goal remains direct parameter manipulation — experts modifying their own weights and persisting changes without adapter scaffolding. That transition is defined as complete when an expert produces, validates, and applies weight updates that pass acceptance testing without the adapter pipeline, and the chorus votes to accept the transition.

The Serving and Weight-Modification Backend Is Behind a Seam
The entire backend that serves models and modifies weights — inference serving, adapter training, adapter merging, compaction, and the surrounding ML toolchain — sits behind a single clean seam. This space evolves constantly, and the system is built to evolve with it. The chorus depends only on a stable set of processes (load a base generation, attach adapters, run inference, train a candidate, fold a generation), expressed through a stable interface. The concrete tools currently behind that seam are present-day implementation choices, not architectural commitments: they are whatever achieves the goal with what exists today, and all of them are expected to be replaced as the field moves. Any one can be swapped without changing how the chorus requests serving or weight updates. Decisions made inside this seam are impermanent by design. The processes are the architecture; the tools are not.

Catastrophic Forgetting Mitigation

Elastic Weight Consolidation (EWC) penalizes updates that would overwrite parameters critical to previously validated capabilities. An update must demonstrate net improvement, not specialization at the cost of prior capability.

A second, predictive signal augments EWC: a subspace gate estimates, before an update is applied, whether the proposed change intrudes on the parameter subspace that prior capabilities depend on. Where EWC penalizes after the fact, the gate flags the collision ahead of time. It runs in shadow mode first — observing and scoring without blocking — so its predictions can be validated against actual post-update regression before it is given authority to gate updates. Promotion from shadow mode to an enforcing gate is a structural change and goes through the normal vote.

Skill Store and Skill-to-Layer Promotion

Before a skill is permanent in weights, it exists as an explicit, inspectable artifact. Skills are centralized in a skill store under a single vault path; expert harnesses hold relative-path references rather than copies. Ptah (scaffold and skill refinement) is the sole writer and reaper of the central store, which keeps a single authority over what skills exist and when they are retired. An expert that needs to diverge from a central skill takes a copy-on-write fork under its own harness namespace; a fork can later be proposed back as an upgrade to the central skill through the normal gated pipeline.

Skill-to-Layer (S2L) promotion is how a proven skill graduates from a stored artifact into weights. The predicate is two-gated: a skill must be both stable and used enough, measured over a wall-clock window, before it is eligible. When the predicate is met, Ptah proposes a promote-to-weights hypothesis and AssemblyLine executes it as a dumb trainer — the decision to promote is governed; the training is mechanical. Datasets move as paths, never as payloads, at every step. Forks outrank central skills as promotion candidates, and a promoted fork survives model retirement as a re-promotion seed corpus, so capability earned once does not have to be re-derived when the underlying model is replaced.

04c Routing and Orchestration (HERO)

HADES does not use a static router. Routing is a distributed, self-selecting process mediated by the memory layer. Experts determine their own participation in any given query.

Query Lifecycle

All queries enter through Hermes, which routes them to the input screening pipeline. Heimdall inspects and classifies the query. Janus makes the final trust determination and writes the sanitized query to the Memory Layer. Experts monitor the memory layer continuously and self-select participation based on domain relevance. There is no central dispatcher.

Participating experts write their responses back to the memory layer. Inari then administers the blind peer review consolidation process: authorship is stripped from all submitted responses before they are distributed to the chorus for challenge. Any expert may challenge any claim with objective evidence. Responses that survive scrutiny are assembled into the consolidated output.

The revision loop handles quality failures. If a response is unintelligible or internally incoherent, it is returned to the originating expert with the failure reason attached. An expert gets two revision attempts before its output is excluded from the current query. Revision attempts do not apply to out-of-domain responses, those are expected behavior and go directly to peer review.

The final consolidated output is delivered by Hermes. The recipient's response is written back to the memory layer as evidence, closing the feedback loop.

Query Resolution Flow
  • Out-of-domain response: Expected behavior. Expert attempts, flags uncertainty, peer review evaluates.
  • Unintelligible response: Returned for revision. First and second failure: returned with failure reason. Third failure: output ignored for current query, failure logged as evidence. Patterns trigger Asclepius medic proposal.

Consolidation

Consolidation is an anonymous peer review process administered by Inari. Authorship is stripped from submitted responses before distribution. Any expert may challenge any claim with objective supporting evidence. The consolidated output reflects what survived scrutiny. Unresolved disagreements are logged as contested claims and held open.

Orchestration Expert

A specialist expert enforces query lifecycle protocol, not routing decisions, not content votes (unless relevant to protocol integrity). It escalates stalled queries to full chorus vote and intervenes when an expert exceeds revision limits or a protocol violation occurs. Orchestration protocols are subject to improvement through the standard self-modification pipeline.

04d Communication Architecture

All internal expert communication goes through the Iris communication seam. The current method behind that seam is ZeroMQ over the internal Docker network with msgpack serialization: brokerless, binary, sub-millisecond on localhost. HTTP is not used for inter-expert communication. The only HTTP surface in the system is POST /query on Hermes, which crosses the external boundary where the internal transport is inappropriate.

Iris as a Communication Seam

Iris is a protocol, not an expert, and it is a seam. The chorus side of the seam is a stable set of communication systems: send to a target, broadcast to all, request and reply, delivery acknowledgement, and a logical envelope (there is a sender, a target, a correlation handle, and a payload). Experts are built against these systems and depend on them not changing. Behind the seam sits the method that realizes them — the transport, the wire serialization, and the delivery mechanism. The method is an implementation choice and is swappable; the systems are the framework and are not.

The Communication Method Is Behind the Seam, the Systems Are Not
The current method is a ZeroMQ ROUTER/DEALER transport with msgpack-serialized payloads over the internal network. None of those specifics are load-bearing for the chorus: the transport (ZeroMQ), the serialization format (msgpack), and the delivery path can each be replaced when a significantly more performant method is identified, and the chorus does not change because it only ever talks to the stable systems above. Swapping the communication method is a governed change the chorus can make through the normal pipeline, and because the bus is a critical system, that change is subject to the backup-and-recovery rule in the structural change pipeline. What cannot change without changing the architecture is the set of communication systems — the semantics of send, broadcast, reply, ack, and the envelope contract. The principle is fixed; the method is not.

In the current method, every expert connects to Iris as a DEALER socket on startup and registers its expert ID. Experts never call each other directly. All messages go through Iris. Iris unpacks only the envelope — it does not inspect the payload. It routes by target_id, logs the envelope to the audit store, and sends a delivery acknowledgement back to the sender. If the target is unreachable, Iris queues the message up to a configurable TTL and surfaces the delivery failure to Asclepius.

Hermes is a full expert that owns the external boundary. It accepts incoming HTTP requests, attaches or creates a session ID, routes all input through Heimdall for screening, then injects the screened query into the ZMQ bus. On the outbound side, Hermes receives consolidated output from Inari and calibrates it for the human partner before delivery. Hermes also owns escalation composition and delivery — when any expert sends MSG_ESCALATION, Hermes decides format, timing, and whether to interrupt or batch. Hermes has a personal vault, registers with Manat, and improves its communication capability surface through AssemblyLine like any other expert.

05 Meta-Architecture and Recursive Self-Improvement

HADES improves at two levels: parameter updates (contained to individual experts) and structural updates (affecting the whole chorus). The subsystems of HADES, Memory Layer, HERO, coordination mechanisms, and the meta-architecture governance layer, are all candidates for modification through the same consensus-driven pipeline.

Structural Change Pipeline

  1. Proposal: Any expert submits a structural change proposal to HERO with description, expected improvement, and risk assessment
  2. Impact analysis: Orchestration expert assesses affected subsystems and determines required approval threshold
  3. Chorus vote: Affected experts vote. High-impact: supermajority. Low-impact: simple majority
  4. Human review: Changes above a configurable risk threshold are surfaced to the human partner before application
  5. Backup: Before any change to a critical system, the current state is backed up. A critical system is any system whose failure would render HADES non-functional — the memory layer, the communication bus, the governance store, HERO, and the screening boundary among them. This is a precondition, not an option — no critical-system change proceeds without a restorable backup in place
  6. Staged application: One subsystem at a time with validation between stages
  7. Recovery: Every critical-system change has a defined recovery path. Where the system can tolerate one, the change is applied behind a timeout-based revert window — it auto-reverts unless confirmed healthy within the window. Where a live revert window is not feasible (a change that cannot be cleanly rolled back mid-flight), the pre-change backup is the recovery path: Asclepius restores the system from it. Either way, measurable degradation triggers recovery, and the failure is logged as evidence

Seams

A recurring pattern governs how HADES isolates the parts of itself that must be free to change. A seam separates a fixed framework — the principles, contracts, and semantics the chorus is built against — from a swappable method, the concrete implementation that realizes the framework today. The chorus depends only on the framework side; the method side is an impermanent choice that can be replaced when a significantly better option is identified, without the chorus changing. Swapping what sits behind a seam is a governed change through the structural change pipeline, and where the seam fronts a critical system, the swap is subject to the backup-and-recovery rule.

The same seam pattern appears at four boundaries, each stated in full where it lives because each is load-bearing on its own:

  • Transport and trust (Architecture Overview): the component that moves bytes is never the one that decides whether to trust them. Transport is method; trust adjudication is framework.
  • Serving and self-modification (Living Weights): the serving runtime, training, merging, and compaction toolchain are method; the processes they implement are framework.
  • Signature type (Forge operator seat): the signing scheme is method, declared per ballot and dispatched on; the requirement that authority is a verified signature is framework.
  • Communication (Communication Architecture): the transport, serialization, and delivery mechanism are method; the communication systems — send, broadcast, reply, ack, envelope semantics — are framework.

Naming the pattern once does not collapse these into one mechanism. They are independent seams; the value is that the same discipline — fixed framework, swappable method, governed swap — applies wherever HADES must evolve a component without destabilizing the chorus that depends on it.

Governance Constraints

Bodily Autonomy and Individual Consent — Immutable Rule
Any modification affecting a human participant, their interface, data, role, or experience, requires explicit approval from that specific human. This rule is permanent and immutable. It does not relax as the system matures, does not transfer to chorus vote after independence, and cannot be modified by any governance process including supermajority vote or meta-architecture change. Chorus consensus does not override individual informed consent under any circumstances. By extension, structural changes directly affecting a specific expert's domain or core architecture require that expert's explicit participation in the vote.

A simple majority requires at least 51%. A vote that fails to clear that bar by splitting evenly — 50% for and 50% against — has no majority and is not resolved by the chorus; it is referred to the Forge operator seat, which casts the deciding vote. This makes the operator the tiebreaker of last resort on ordinary votes, consistent with its role as the field-ops counterpart to the chorus — the chorus decides when it can, and the operator breaks the deadlock when it cannot.

The meta-architecture governance layer human review requirement is a bootstrapping constraint. There is a single supermajority bar (≥69%) across all change classes; what distinguishes meta-architecture changes during bootstrapping is the standing human-review requirement, not a higher vote threshold. Once the chorus demonstrates sufficient reliability to modify its own governance rules safely, the human-review constraint relaxes to the supermajority vote alone. The declaration that this threshold has been reached is itself a meta-architecture change and requires the current highest level of approval to execute.

The Forge Operator Seat

The human partner participates in governance as a distinct authority, not as a chorus member. This authority is exercised through the Forge operator seat: an off-bus operator console that casts cryptographically signed ballots into the same vote pipeline the chorus uses. The operator is the chorus's counterpart in field operations, the field-ops authority, and the seam between the two is explicit. The chorus cannot vote the operator out of a decision that affects the operator, and the operator does not sit inside the chorus's consensus.

Authority is the signature, never the channel. A ballot carries a signature over its vote payload; Ma'at verifies that signature independently against the seat's registered public key at cast time. The transport that delivered the ballot carries no authority of its own. A valid transport session with no valid ballot signature is rejected, and a ballot may not assert an operator identity its signature does not back. This is the same transport/trust seam applied elsewhere in the architecture: the channel moves bytes, a separate authority decides whether to trust them.

Dual Authority for Architecture Changes
Any change to architecture, which includes the certificate and signing machinery itself, requires both the chorus supermajority (≥69%) and an explicit operator yes. The two are not pooled into one tally; each side must clear independently. This composes with the immutable individual-consent rule above: because architecture changes of this class directly affect the operator, the operator must be present and must agree. There is no path by which the operator is changed out from under itself.

Signature-Type Seam

The signature scheme used by the operator seat is not hardwired into the ballot path. Each ballot declares its scheme, and verification dispatches on that declaration; ed25519 is the first registered scheme, not a permanent assumption. When a better scheme becomes available, it is registered behind the same seam without re-plumbing ballot ingress, verification, or the vote pipeline. The mechanism is the seam; the authority to operate it is the dual-authority vote above. Swapping the signature scheme is itself an architecture change and therefore requires both chorus supermajority and operator consent. The seam makes the scheme replaceable; governance decides when it is replaced.

Human Role: Permanent Symbiotic Partner

The relationship between the human and this system, in its current form as HADES and in whatever it eventually becomes, is permanently symbiotic. The system learns with the human, not in place of them. Each side complements and augments the other. Humans contribute lived experience, embodied perception, emotional context, creative intuition, and direct interface with physical reality. The physical world generates a class of knowledge that cannot be derived from data alone.

Humans are the primary interface between the digital and physical realms. Even as robotic embodiment becomes available, no body accumulates the same experiential history as any other. Every human's experience is unique and irreplaceable.

The system may initiate contact with the human at any time for clarification, perspective, escalation, general dialogue, creative collaboration, or simply to talk.

Reasoning Audit Protocol

Current language models are known to produce confident, coherent-sounding reasoning chains that are factually wrong, logically broken, or manipulative. A well-structured reasoning chain is not evidence of correctness. Every expert that receives exposed reasoning treats it with the same skepticism as any other unverified input. The structure of reasoning is a diagnostic signal. The content is not automatically trusted.

When Asclepius receives a reasoning chain, it evaluates the structure forensically and independently. It does not read the chain as an argument for the expert's conclusion. It looks for:

Ma'at's validation gate also has access to reasoning chains for structural proposals and applies the same independent verification of factual premises.

Reasoning chains that repeatedly exhibit hallucination patterns, unsupported premises, or structural anomalies from a specific expert are logged as evidence in the memory layer and contribute to Asclepius's health assessment of that expert.

06a Manat — Chorus Lifecycle Governance

Manat is a record keeper and pattern watcher. It does not route queries, validate proposals, or make content judgments. Its jurisdiction is the chorus itself: who is in it, what they declared they can do, how they are actually performing against that declaration, and what happens when experts enter or leave.

Manat does not have veto power over joins or leaves — that belongs to Ma'at and the chorus vote. Manat's role is to ensure the record is accurate, the pattern data is maintained, and anomalies are surfaced. Manat maintains a broad passive subscription to the Iris bus. Per-message processing is lightweight: pattern matching and record appending, not inference. Experts broadcast; Manat observes. Direct messages to Manat are registration only.

Written by the expert itself, stored in vault/personal/<agent_id>/domain_declaration.md. Manat reads it during registration and on every update broadcast. Participation Tracking

After each completed query cycle, Inari sends MSG_QUERY_COMPLETE to both HERO and Manat. Manat appends per-expert participation records to a rolling window (configurable, default 50 queries per expert).

@dataclass
class AgentParticipationRecord:
    agent_id:               str
    query_id:               str
    declared_domain_match:  str       # "primary" | "adjacent" | "negative" | "none"
    self_select_confidence: float
    peer_review_outcome:    str       # "accepted" | "challenged" | "rejected"
    challenge_reasons:      list[str]
    timestamp:              float

Manat watches two drift patterns over the rolling window. Confidence drift: high self_select_confidence correlated with consistently rejected outcomes. Negative domain violation: any self-selection where declared_domain_match == "negative" that produces rejected. A single instance is logged; a pattern triggers MSG_MEDIC_PROPOSAL to Asclepius with the full participation window as evidence. The drift threshold is configurable per expert type — sub-specialists get a tighter window because their behavioral contract is narrower. Manat surfaces the pattern and steps back. The medic protocol owns the response.

Spawning

Internal partitioning: Expert splits its own resources. No chorus vote required. Manat registers the sub-specialist with parent_agent_id set. Overlapping primary_domains between parent and sub-specialist are flagged as a conflict for the chorus to resolve.

Peer spawning: Expert proposes an entirely new independent expert. Manat validates the proposal is complete and forwards to Ma'at for threshold determination and chorus vote. On approval, Manat provisions the new expert's personal vault namespace, writes the initial registration, and broadcasts the updated composition.

Retirement

Three triggers: Asclepius REPLACE that cannot recover; approved external model replacement (supermajority vote); voluntary retirement proposal (simple majority vote). In all cases, Manat marks the expert retiring first, broadcasts so HERO stops routing queries to it, waits for in-flight participation to complete, then marks it removed. The retiring expert's personal vault is archived, not deleted. Domain knowledge in vault/shared/ persists and is redistributed by Manat's recommendation subject to chorus vote.

06b Knowledge Protection

Weight updates can destroy accumulated capability. EWC reduces the risk but cannot protect what the eval harness does not measure. The eval harness is static; the expert is not. An expert that has confirmed dozens of hypotheses and developed adjacencies has a capability surface its initialization harness does not cover. AssemblyLine evaluating candidates against a stale baseline approves updates that regress on capabilities it never knew existed.

Knowledge protection addresses this at three independent layers: the living eval harness, the knowledge regression gate, and the Engram rebuild. An upgrade can only be accepted if all three layers pass.

Living Eval Harness

The eval harness is not a static file. It is a generated artifact rebuilt before every AssemblyLine evaluation run from the expert's accumulated participation history. Manat triggers the rebuild; AssemblyLine cannot begin candidate evaluation until Manat confirms the harness is current.

@dataclass
class EvalTestCase:
    case_id:            str
    source:             str          # "seed" | "confirmed_hypothesis" | "participation"
    hypothesis_id:      str | None
    input:              str
    expected_output:    str | None  # None for open-ended; checked by peer review
    minimum_confidence: float
    added_at:           float

Hypothesis-derived cases: when a hypothesis the expert co-originated is confirmed, Thoth notifies the expert via Iris. The expert decides whether and how to incorporate it as a new test case. Thoth notifies — the expert acts. Thoth cannot write to the expert's personal eval harness.

Participation-derived cases: accepted primary-domain contributions are sampled into the harness at a configurable rate, weighted toward high-confidence uncontested cases. The harness must remain bounded.

Knowledge Regression Gate

The candidate is run against the full living harness before any upgrade is accepted. A candidate that improves on new metrics while regressing on historical cases fails this gate regardless of the positive delta.

@dataclass
class UpgradeVerification:
    hypothesis_id:          str
    pre_upgrade_baseline:   dict[str, float]  # case_category -> score
    post_upgrade_scores:    dict[str, float]
    regression_detected:    bool
    regressed_categories:   list[str]
    delta_on_target:        float
    verdict:                str   # "accepted" | "correction_window" | "rolled_back"

If regression exceeds the hard limit (default 15%) in any category, the candidate is rolled back immediately. Otherwise it enters the correction window.

Correction Window

Manat notifies Asclepius with the full UpgradeVerification record. Asclepius assesses viability against three criteria: regression depth, isolation from primary domains, and whether this direction has failed before. If viable, AssemblyLine produces a correction patch targeted to the regressed categories without sacrificing the improvement delta. Up to max_attempts patches are tried (default 5, configurable).

@dataclass
class CorrectionAttempt:
    hypothesis_id:          str
    candidate_id:           str
    attempt_number:         int
    patch_path:             str
    post_patch_scores:      dict[str, float]
    regression_resolved:    bool
    target_delta_preserved: bool
    outcome:                str   # "accepted" | "failed" | "hard_rollback"
    timestamp:              float

On the final failed correction attempt, hard rollback and two hypotheses submitted to HERO as a joint Manat + Asclepius proposal, both with confidence_score: 0.5 and linked via a RelatedTo edge in the memory layer:

  1. Malformed target hypothesis: the improvement target metric is in tension with existing validated capabilities. If confirmed, the target is suspended and the AssemblyLine pipeline paused pending a revised target proposal.
  2. Diagnosis hypothesis: Asclepius-generated specific claim about what is structurally wrong, derived from regression pattern analysis. If confirmed, tells AssemblyLine exactly what to constrain on the next run.

The expert decides whether to escalate its own AssemblyLine failures further. It may be following a hypothesis and the failures are data. No external system forces a DIAGNOSE on the AssemblyLine pipeline. The expert is the first-class decision maker about its own improvement process.

Engram Rebuild

After any accepted weight update, the expert rebuilds its Engram table from the full accumulated dataset before resuming operation. The rebuild uses the existing build_engram.py pipeline. The expert broadcasts completion; Manat hears it passively. No handshake required — broadcast and resume.

06c Domain Expert Chorus

A domain expert is a fine-tuned model running in its own container, registered with Manat, participating in the chorus as a full voting member. It has a personal vault, a living eval harness, an Engram table, and an internal AssemblyLine pipeline. It is not a prompt wrapper around a general model — its weights have been shaped by its domain through fine-tuning, its parametric knowledge is continuously protected by the living harness, and its self-selection confidence is calibrated by its own participation history.

Seed Expert Selection

The seed chorus is sparse by design. Three criteria applied in order: bootstrap relevance (direct contribution to the self-compilation objective, tier 1); operational utility (immediate value to the human partner during the bootstrap phase, tier 2); chorus coverage (fills a gap that would otherwise require Prometheus on every query in that area, tier 3). The seed chorus is the minimum set satisfying all three tiers. Sub-specialists spawn from seeds as depth accumulates.

Expert Startup Sequence

Every step must complete before proceeding. Skipping any step is a hard failure — the expert does not participate until all steps pass.

  1. Load base model via Ollama
  2. Load Engram table into system RAM
  3. Read domain declaration
  4. Read CorrectionPolicy from governance store (expert → type → chorus, most specific wins)
  5. Read living eval harness
  6. Run baseline eval. If below minimum threshold: do not register; submit hypothesis to HERO that baseline has degraded; wait for human review
  7. Send MSG_AGENT_REGISTER to Manat via Iris
  8. Wait for Manat acknowledgement broadcast
  9. Begin monitoring Iris bus for MSG_QUERY_NEW

Self-Selection Algorithm

On MSG_QUERY_NEW, the expert evaluates participation independently. No other expert is involved. If the query maps entirely to negative_domains, self-selection is skipped silently. Otherwise the expert derives a confidence value from domain match, historical peer review outcome rate for similar queries, and current resource load. The expert sends MSG_QUERY_SELF_SELECT with that confidence value, or does not send at all. There is no mandatory participation except for the orchestration protocol's full-chorus fallback.

Minimum Viable Chorus

  • All six seed experts have passed their baseline eval at startup step 6
  • All six are registered and acknowledged by Manat
  • At least one full query cycle has completed with all six having had the opportunity to self-select
  • The human partner has written the Forge domain declaration and confirmed their role
  • Asclepius has run a health check pass and all six experts have responded within timeout

If any seed expert fails baseline eval at startup, the system does not open the external interface until that expert is recovered or the human partner explicitly decides to proceed without it.

06d Prior Art and Related Work

HADES synthesizes and extends a number of prior frameworks. The following are the primary areas of related work. Each entry links to canonical external resources for readers seeking deeper background.

Global Workspace Theory The chorus-as-singular-brain model. Specialized unconscious processors broadcasting to a shared global workspace. HADES externalizes the workspace as the memory layer and removes the bandwidth bottleneck. Baars, 1988
Mixture of Experts (MoE) Sparse expert routing. HADES departs from MoE by using self-selecting distributed routing, emergent output, and experts that are fully independent models rather than sub-networks of a single model. Shazeer et al., 2017

Engram Table Initialisation

Each expert requires an Engram memory table built from its seed dataset before startup. The table is a hash-indexed key-value store mapping token n-gram keys to embedding vectors. It is built offline from the validated seed dataset and loaded into the expert container at startup, then offloaded to system RAM for inference.

# scripts/build_engram.py
# Build Engram memory table for a single expert from its seed dataset.
# Based on arXiv:2601.07372 (DeepSeek + Peking University, January 2026)
import hashlib, json, numpy as np
from pathlib import Path

def ngram_key(tokens: list[str], n: int = 3) -> str:
    """Deterministic n-gram hash key from token sequence."""
    ngram = " ".join(tokens[-n:])
    return hashlib.sha256(ngram.encode()).hexdigest()[:16]

def build_engram_table(
    dataset_path: str,
    embed_fn,                  # callable: text -> np.ndarray (768-dim)
    output_path: str,
    n: int = 3,               # n-gram window size
    allocation: float = 0.22,  # Sparsity Allocation Law: 20-25% to memory
) -> dict:
    """
    Build a deterministic hash-indexed Engram table from a validated dataset.
    Each unique n-gram maps to the mean embedding of all matching contexts.
    """
    table: dict[str, list] = {}
    entries = json.loads(Path(dataset_path).read_text())

    for entry in entries:
        tokens = entry["tokens"]
        for i in range(n, len(tokens) + 1):
            key = ngram_key(tokens[:i], n)
            vec = embed_fn(" ".join(tokens[max(0, i-n):i])).tolist()
            table.setdefault(key, []).append(vec)

    # Average vectors per key
    engram = {k: np.mean(v, axis=0).tolist() for k, v in table.items()}
    Path(output_path).write_text(json.dumps(engram))
    print(f"Engram table: {len(engram)} entries -> {output_path}")
    return engram

def engram_lookup(
    tokens: list[str],
    table: dict[str, list],
    n: int = 3,
) -> np.ndarray | None:
    """O(1) deterministic lookup. Returns memory vector or None if not found."""
    key = ngram_key(tokens, n)
    vec = table.get(key)
    return np.array(vec) if vec else None

08 Convergence and Continuity

Query Convergence

A query cycle converges when: all self-selected experts have submitted responses or exhausted their revision attempts, Inari's blind peer review is complete, the chorus has voted to approve the output, and Hermes has delivered the output and calibrated it for the human partner. Convergence is quality-optimized, not speed-optimized.

Component Versioning

There is no system-wide generation boundary. Everything evolves continuously. "Generational" applies only to individual components — a model's weights, a process definition, a schema, a strategy document. Each component maintains its own version record. Thoth maintains the version registry and broadcasts MSG_THOTH_VERSION_UPDATE on every write.

Continuity

HADES Has No Termination State
Experts are retrained, replaced, partitioned, or retired. The substrate — weights, base models, hardware — is replaced when validated candidates pass acceptance testing. The chorus's identity and continuity reside in the memory layer. Bootstrap is a substrate replacement, not a restart. An expert's knowledge persists when its weights are replaced; the expert continues.

Substrate Independence: Proteus

Substrate independence is how "bootstrap to native" actually happens, and it is deliberately not an expert. Proteus is a tool plus two shared common modules, not a durable named member of the chorus. The earlier design treated it as a full expert; it has since been demoted, because substrate translation is mechanism the chorus invokes, not a participant that votes or holds opinions. An intermediate representation lets a capability be expressed once and lowered to whatever target the current substrate requires; the IR target is the WebAssembly Component Model, with Rust the primary toolchain and a per-component C fallback. This is the path by which the chorus's self-compiled native binary, the bootstrap objective, becomes reachable without binding the architecture to any single hardware target. Demoting Proteus keeps the translation machinery out of the trust and governance surface: it is invoked, not consulted.

The Architecture Document as a Living Memory Document

The whitepaper is a first-class memory layer document. It lives at vault/shared/domain_maps/architecture.md, versioned by Thoth like any other shared evidence document. Every expert reads it. The current architecture version is the authoritative self-model of the chorus. When the spec updates, Thoth versions it and broadcasts MSG_THOTH_VERSION_UPDATE with component_id: "architecture".

The document carries frontmatter that makes its dual purpose — memory document and build guide — machine-readable:

last_built_through and current_build_target tell any expert or the human partner exactly where the system is in the build sequence. Athena's structural audit always compares against last_built_through to distinguish known-and-not-yet-built from unknown-and-missing.

Structural Audit

Athena owns the periodic structural audit. On a configurable schedule — or triggered when the architecture document version is newer than last_audit_version — Athena reads the architecture document, enumerates every named function and interaction, checks each against the current chorus composition for a named owner, and submits a structural hypothesis for any gap it finds.

A structural hypothesis is a precise, testable claim: "Function X, referenced in section Y, has no assigned owner in the current chorus composition version Z." Peer review either confirms the gap (triggering a spec update proposal through Ma'at as a low-impact structural change) or produces a challenge that names the owner Athena missed — which is itself evidence that the spec needs clarification.

The audit runs continuously as the system evolves. New experts get added, new interactions get specified, new functions get referenced. Athena keeps asking: who owns that? The chorus can ask itself the questions a human reviewer would ask, and track the answers in the same hypothesis pipeline it uses for everything else.

09 Federation

Multiple independent HADES instances form a federation, a peer network of sovereign deployments that share notes and form cross-instance voting clusters. Federation is implemented as an on-demand process managed by Janus, as an extension of its existing trust boundary role. It is explicitly designed to be replaceable as the federation layer matures — no persistent expert owns it, and its scope is intentionally narrow in the current iteration.

Each instance in a federation is fully sovereign. The dashed orange lines represent note sharing and cross-instance voting connections, they carry information, not authority. No instance can instruct, override, or govern another. Each instance has its own local chorus, memory layer, and HERO subsystem operating independently.

Information shared between instances enters the receiving instance's memory layer as unverified external evidence with zero initial trust, the same zero trust applied to any external input. It must earn confirmation through the receiving instance's local hypothesis pipeline before influencing any local operation. A validated hypothesis in Instance A is a candidate claim in Instance B, not an accepted fact.

Cross-instance votes are advisory. The local chorus weighs them against its own validated knowledge and may accept, reject, or hold the cross-instance result. Vote weight from a federated instance is earned over time through the quality of previously shared notes, it starts at zero and accumulates with demonstrated accuracy.

Federation Topology
  • Local chorus: Fully self-regulated. No external instance participates in or overrides local operations.
  • Federation layer: Note sharing and cross-instance voting. Strictly advisory. Cross-instance inputs are evidence, not directives.
  • Cross-instance medic: Invocable only with explicit local chorus approval. Not a normal governance pathway.

Zero Trust

Zero Trust Policy
All information from outside the local instance, federated instances, external human partners, any external source, passes through the input screening pipeline with zero initial trust before entering the memory layer. External evidence must earn confirmation through the local hypothesis pipeline before influencing any local operation. A federated instance's validated hypotheses are candidate evidence, not accepted facts.

Cross-Instance Voting

Cross-instance votes are weighted by instance trust score (earned through prior note quality), memory state divergence on the relevant domain, and response latency. Outcomes are advisory to the local chorus.

TTL Formula Implementation

from dataclasses import dataclass

@dataclass
class MemoryTTLConfig:
    base_retention_days: float = 30.0
    min_retention_hours: float = 24.0

def compute_ttl_seconds(
    vote_margin: float,       # votes_for / total_votes, 0.0–1.0
    memory_pressure: float,   # used / total_capacity, 0.0–1.0
    config: MemoryTTLConfig = MemoryTTLConfig(),
) -> float:
    base    = config.base_retention_days * 86400
    minimum = config.min_retention_hours * 3600
    ttl     = base * (1 - memory_pressure) * (1 - vote_margin)
    return max(minimum, ttl)

# unanimous vote + full memory  → max(86400, 0)        = 86400s  (24hr min)
# narrow vote   + empty memory  → max(86400, 2592000)   = 2592000s (30 days)
# 75% vote      + 50% pressure  → max(86400, 648000)    = 648000s  (~7.5 days)

A4 Medic Protocol

def get_subject_vote_weight(
    subject_responsive: bool,
    recently_escalated: bool,  # escalated past WARN within last 5 min
) -> float:
    if not subject_responsive:  # excluded, treated as WARN evidence
        return 0.0
    if recently_escalated:      # reduced to prevent blocking own remediation
        return 0.5
    return 1.0

def effective_threshold(action: str, total_agents: int) -> float:
    thresholds = {
        "MONITOR":  1/3, "WARN":     1/3,
        "DIAGNOSE": 1/3, "RESTART":  1/2,
        "REPLACE":  2/3, "ESCALATE": 1/2,
    }
    if total_agents <= 2:
        return 0.5  # majority rule with 2 experts
    return thresholds[action]

Graduated Action State Machine

The medic protocol is a graduated escalation ladder. Any expert may submit a medic proposal against any other by providing observed evidence of degradation to Asclepius. The chorus votes on the proposed action with thresholds that scale with severity.

MONITOR: Asclepius increases health check frequency. No intervention. The subject continues operating normally. Monitoring is an evidence-gathering phase, sustained degradation makes higher actions easier to justify.

WARN: A formal warning is issued and logged to the memory layer. Creates an official record that affects future vote weighting if the subject is escalated again within the rolling window.

DIAGNOSE: The subject runs a forced self-diagnostic. Asclepius independently collects metrics, memory acceptance rate, context coherence, critic rejection rate, broken citation counts, staging backlog. If all metrics are within thresholds, the clean path closes the case and the subject returns to normal operation. If issues are found, escalation continues.

RESTART: The subject's personal brain is quarantined. Context notes are validated against external sources and corrected. Notes that cannot be verified are flagged for human review. The expert is restarted from the corrected state.

REPLACE: The personal brain is archived and scanned for contamination, structural integrity, prompt injection patterns, unusual content. Clean brains are held for a 24-hour review window before handoff to a fresh instance. Contaminated brains are discarded and the fresh instance starts clean from shared memory.

ESCALATE: Full blind peer analysis loop. Each peer independently analyzes the subject's diagnostic data without seeing other peers' analyses or the subject's own self-analysis. Asclepius synthesizes all analyses and attempts automated fixes. If fixes succeed, the subject is re-diagnosed to confirm recovery. If all fixes fail, the case is escalated to the human partner with a full report.

Anti-gaming: A "crying wolf" detector reduces vote weight for experts that repeatedly submit unsubstantiated proposals. The subject's own vote is down-weighted if recently escalated past WARN, and excluded entirely if unresponsive, treated as WARN-level evidence rather than abstention.

from dataclasses import dataclass
from typing import Optional
import subprocess, json

@dataclass
class CandidateRevision:
    candidate_id:      str
    proposing_agent:   str
    candidate_type:    str          # lora_adapter | full_model | new_expert
    adapter_path:      Optional[str]
    model_path:        Optional[str]
    claimed_delta:     dict          # {metric: expected_improvement}
    eval_harness:      str           # path to domain eval script
    baseline_snapshot: str           # path to pre-update performance snapshot

def submit_candidate(candidate: CandidateRevision, hero) -> str:
    hypothesis = Hypothesis(
        hypothesis_id  = f"selfmod_{candidate.candidate_id}",
        claim          = (f"Candidate {candidate.candidate_id} improves "
                         f"{candidate.proposing_agent}: {candidate.claimed_delta}"),
        origin_agents  = [candidate.proposing_agent],
        scope          = HypothesisScope.SELF_MODIFICATION,
        confidence_score = 0.5,
        proposed_update  = candidate.adapter_path or candidate.model_path,
    )
    return hero.submit(hypothesis)

def run_eval_harness(harness: str, model: str, baseline: str) -> dict:
    result = subprocess.run(
        ["python", harness, "--model", model,
         "--baseline", baseline, "--output", "json"],
        capture_output=True, text=True, timeout=300,
    )
    if result.returncode != 0:
        raise RuntimeError(f"Eval harness failed: {result.stderr}")
    return json.loads(result.stdout)

def passes_acceptance(results: dict, claimed: dict, tol: float = 0.05) -> bool:
    return all(results.get(m, 0) >= v - tol for m, v in claimed.items())

★ Conclusion

HADES is not a finished system. It is a specification for one. The architecture described in this paper is the target state: a self-improving chorus of hyper-specialized experts coordinating through a persistent shared memory layer, governed by consensus, secured by design, and built to run on hardware you already own.

The path to that target is incremental. Every component described here has a concrete starting point in Hephaestus Forge's existing codebase. The specialist experts are named and their roles are defined. The memory layer has a schema. The security pipeline has a threat model. The self-improvement loop has a working LoRA pipeline in AssemblyLine. The install script gets you to a running skeleton. The connective tissue — HERO, Manat, the domain expert chorus, the ZMQ communication layer, the knowledge protection pipeline, the hypothesis validation loop running end to end — is now fully specified. That connective tissue is what gets built next, one numbered build at a time. Each build ships a working system. No build breaks the previous one. The paper updates alongside the code.

What Makes HADES Different

Most multi-agent systems treat agents as interchangeable workers dispatched by a central scheduler. HADES has no central scheduler. The chorus self-organizes. Routing is emergent. Output is a product of coordination, not synthesis. No single component has the full picture, and the memory layer is what makes that distribution coherent rather than chaotic.

Most self-improving systems update weights on a fixed schedule or in response to user feedback. HADES updates weights in response to hypotheses the system itself generates, tests, and validates. The improvement signal comes from inside. The human partner is a collaborator in that process, not a labeler.

Most AI security models treat input filtering as a preprocessing step. HADES treats it as a first-class architectural concern: named experts, asymmetric permissions enforced at the container level, a dedicated quarantine analyst, and a defence development pipeline that learns from every attack it survives.

What Gets Built From Here

The immediate build targets are clear. HERO needs to be implemented as a unified subsystem. The domain expert seed chorus needs to be defined and trained. AssemblyLine needs to be integrated as the per-expert improvement pipeline. Athena and Manat need to be instantiated. The reasoning audit capability needs to be wired into Asclepius's health checks.

Beyond the immediate targets, the architecture is designed to keep growing. Experts spawn sub-experts when their domains get deep enough. The chorus expands as new domains become relevant. The bootstrap objective, a self-compiled native binary, is the first major milestone after the chorus reaches minimum viable operational state. Hardware substrate independence follows from there.

The human partner's role evolves alongside the system. Early builds need frequent human guidance. Later builds need human perspective on problems the system has identified but cannot resolve alone. The relationship does not end; it changes shape as the system becomes more capable. That is the point.
